# V1.6 — Action Card Polish + Documents Section

**Date:** 2026-04-11
**Status:** Approved (pending implementation plan)
**Scope:** Action card UX completion, audit trail visibility, and agent-generated documents

---

## 1. Goal

Two demo-ready features that close the gap toward the Okra AI CMO target shape:

1. **Action card polish** — Add the Approve button to all non-AR card types, show audit history inline on every card, and render a visual badge for resolved statuses. The `ActionEvent` table already exists and the PATCH route already writes events — this work surfaces them in the UI.

2. **Documents section** — A new `/documents` page where the agent generates and stores narrative reports (Monthly Variance Report, AR Aging Summary). Reports auto-generate on CSV upload and can be manually regenerated from the Documents page.

### Out of scope
- Email delivery / SMTP integration
- PDF export of documents
- Document versioning or diff
- Scheduled/cron-triggered report generation (V2)

## 2. Feature 1: Action Card Polish + Audit Trail

### 2.1 Current state

- `ActionCard` already has `onApprove` prop and renders an Approve button for non-AR pending cards
- `ActionEvent` model exists in Prisma with `fromStatus`, `toStatus`, `actionId`, `userId`, `createdAt`
- PATCH `/api/actions/[id]` already creates `ActionEvent` rows on every status change
- Resolved-state rendering exists (approved/flagged/dismissed labels) but no audit history is visible

### 2.2 Changes

#### API: GET `/api/actions/[id]/events`

New route that returns the audit trail for a single action:

```ts
// Response shape
{
  events: Array<{
    id: string;
    fromStatus: string;
    toStatus: string;
    createdAt: string; // ISO
  }>
}
```

Query: `prisma.actionEvent.findMany({ where: { actionId }, orderBy: { createdAt: "asc" } })`.

#### UI: ActionCard audit trail toggle

Add a small "History" button (clock icon) at the bottom of every card that has at least one event. Clicking it expands a compact timeline:

```
  pending → approved   2 min ago
  pending → flagged    5 min ago
```

Implementation:
- New state: `showHistory` boolean, `events` array (lazy-loaded on first toggle)
- Fetch from `GET /api/actions/[id]/events` on first expand
- Render as a `<ul>` with `text-xs` items showing `fromStatus → toStatus` and relative time
- Cache events in component state; no re-fetch unless card status changes

#### UI: Status badges for resolved cards

Current resolved-state rendering is correct. No changes needed — just verify it works for all action types including AR cards after send/snooze/escalate.

## 3. Feature 2: Documents Section

### 3.1 Data model

New Prisma model:

```prisma
model Document {
  id           String   @id @default(cuid())
  userId       String
  type         String   // "variance_report" | "ar_summary"
  title        String
  body         String   // Full markdown text
  dataSourceId String?  // Which upload triggered it (null for manual)
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  user       User        @relation(fields: [userId], references: [id])
  dataSource DataSource? @relation(fields: [dataSourceId], references: [id])
}
```

Additions to existing models:
- `User`: add `documents Document[]`
- `DataSource`: add `documents Document[]`

Migration: `add-document-model`

### 3.2 Agent tools (2 new)

Added to `createFinancialTools(userId)` in `lib/agent/tools.ts`:

#### `generate_variance_report`

**Input:** `{ dataSourceId?: string }` (optional — if omitted, uses all ready data sources)

**Behavior:**
1. Reads `FinancialRecord` rows (all, or filtered by dataSourceId)
2. Reads `Action` rows where type = "variance"
3. Returns a structured data payload for the agent to compose a narrative report
4. The agent writes the narrative; the tool returns the data

**Output:** JSON with records summary, top variances, category breakdown, action counts by severity

#### `generate_ar_summary`

**Input:** `{ dataSourceId?: string }`

**Behavior:**
1. Reads `Invoice` rows with status/age info
2. Reads `Action` rows where type = "ar_followup"
3. Returns structured data: total outstanding, aging buckets, dunning activity, escalation candidates

**Output:** JSON with AR metrics for agent to compose narrative

#### `save_document`

**Input:** `{ type: "variance_report" | "ar_summary", title: string, body: string, dataSourceId?: string }`

**Behavior:** Creates a `Document` row in the database.

**Output:** `{ id, title, createdAt }`

### 3.3 Agent skill update

Update `agent/DUTIES.md` to add report generation duties:
- After variance analysis completes: call `generate_variance_report` to gather data, compose a Monthly Variance Report narrative, then call `save_document` to persist it
- After AR scan completes: call `generate_ar_summary` to gather data, compose an AR Aging Summary, then call `save_document` to persist it

### 3.4 Auto-generation on upload

In `lib/agent/index.ts`:

- `analyzeUpload()` prompt updated to include step: "After creating actions, generate a variance report and save it as a document"
- `analyzeArUpload()` prompt updated to include step: "After creating actions, generate an AR aging summary and save it as a document"

The agent uses the new tools within the same query session — no separate API call needed.

### 3.5 API routes

#### `GET /api/documents`

Returns all documents for the authenticated user, sorted by `createdAt` desc.

```ts
// Response
{ documents: Array<{ id, type, title, createdAt, updatedAt }> }
```

No body returned in list view — just metadata for the sidebar list.

#### `GET /api/documents/[id]`

Returns a single document with full body.

```ts
// Response
{ id, type, title, body, dataSourceId, createdAt, updatedAt }
```

#### `POST /api/documents/generate`

Manual generation endpoint. Request body:

```ts
{ type: "variance_report" | "ar_summary" }
```

Triggers an agent query (similar to `analyzeUpload` but for report generation only). This is a blocking request — the server runs the agent, saves the document, and returns when complete. Returns:

```ts
{ id, title, createdAt }
```

### 3.6 UI: Documents page

**Route:** `app/(dashboard)/documents/page.tsx`

**Layout:** Two-column on desktop, stacked on mobile.
- **Left panel:** Document list — cards showing title, type badge, date. Click to select.
- **Right panel:** Document viewer — renders the selected document's markdown body. Shows a "Regenerate" button at the top.

**Top bar:** "Generate Report" button with dropdown:
- Monthly Variance Report
- AR Aging Summary

Clicking triggers `POST /api/documents/generate`, shows a loading spinner, then adds the new document to the list and selects it.

**Empty state:** "No documents yet. Upload financial data or click Generate Report to create your first document."
