# V1.5 — AR Follow-ups Design

**Date:** 2026-04-10
**Status:** Approved (pending implementation plan)
**Scope:** V1.5 of Lyzr AI CFO — Accounts Receivable follow-up automation

---

## 1. Goal

Add a second agent skill — **AR follow-ups** — on top of the existing variance-review foundation. The agent detects overdue invoices, classifies them by age, creates action cards in the existing feed, and drafts dunning emails the user copies and sends from their own mail client. No outbound email delivery, no reply tracking, no autonomous finalization.

This is the first test of the "composable skills" principle: a new finance workflow added to the same agent without platform changes.

### Scope decision: Option B — Detect + Draft + Track

- Agent detects overdue invoices and buckets by age
- Creates AR action cards in the same feed (new type, same audit/approval model)
- Drafts dunning email per invoice with tone that escalates by days overdue
- User reviews and clicks **Copy & Mark Sent / Snooze / Escalate** inline
- System tracks invoice state so the agent stops re-nagging after "sent" and knows when to escalate
- **Out of scope:** outbound email delivery, SMTP/Resend integration, reply tracking, bounce handling

### Data source decision: CSV-only

User uploads an AR aging CSV. Same pattern as variance CSV ingestion. Google Sheets and QuickBooks integration stay in their own future sprints where they benefit every skill, not just AR.

---

## 2. Architecture Overview

V1.5 is an **additive skill layer**, not a platform change. The existing architecture (Next.js routes → gitclaw agent → Prisma → Actions Feed UI) stays intact. AR composes on top as:

1. **New agent skill** at `agent/skills/ar-followup/SKILL.md` — peer of `variance-review`, same file shape.
2. **New agent tools** in `lib/agent/tools.ts` — four new `tool(...)` exports appended to `createFinancialTools(userId)`. Chat and upload entry points pick them up automatically.
3. **New Prisma model** `Invoice` + optional `invoiceId` and `draftBody` fields on `Action`. `FinancialRecord` stays untouched — variance and AR are sibling domains sharing `User` / `DataSource` / `Action` / `ActionEvent`.
4. **New CSV parser** at `lib/csv/ar-parser.ts` with LLM column-mapping fallback. Upload route auto-detects AR vs variance shape from headers.
5. **Action card variant** — `action-card.tsx` renders AR-specific buttons when `action.type === "ar_followup"` and inline-expands to show the draft email.
6. **Proactive scan entry point** — a "Scan AR" button on the morning briefing. No cron. Scheduler stays V2.

**Principle:** one agent, two skills, one feed, one card component. Reuse existing mechanisms unless the domain genuinely requires new state.

## 3. Data Model

### 3.1 New `Invoice` model

```prisma
model Invoice {
  id             String    @id @default(cuid())
  dataSourceId   String
  invoiceNumber  String    // "INV-1847" — displayed in UI, not the PK
  customer       String    // customer name as string; no Customer table yet
  customerEmail  String?   // nullable — agent degrades gracefully when missing
  amount         Float
  invoiceDate    DateTime
  dueDate        DateTime
  status         String    @default("open")
  //                         open | sent | snoozed | escalated | paid
  lastDunnedAt   DateTime?
  snoozedUntil   DateTime?
  createdAt      DateTime  @default(now())

  dataSource DataSource @relation(fields: [dataSourceId], references: [id])
  actions    Action[]

  @@unique([dataSourceId, invoiceNumber])
}
```

### 3.2 `Action` model additions (non-breaking)

```prisma
model Action {
  // ...existing fields unchanged...
  invoiceId    String?   // NEW — nullable, set only for AR actions
  draftBody    String?   // NEW — cached dunning email, nullable
  invoice      Invoice?  @relation(fields: [invoiceId], references: [id])
}
```

### 3.3 `DataSource` additions

Adds sibling relation `invoices Invoice[]`. No other model touched.

### 3.4 Rationale

- **`Invoice.status`** is the single source of truth for "should the agent nag this again." `scan_ar_aging` filters on it directly.
- **`(dataSourceId, invoiceNumber)` unique index** makes re-upload idempotent — a second upload of the same aging report updates existing invoices rather than duplicating.
- **`draftBody` cached on Action** means we don't re-hit the LLM when the user expands a card, and any future edit flow doesn't lose state.
- **`type = "ar_followup"`** requires no migration since `Action.type` is already a `String`, not a Prisma enum.
- **`customerEmail` nullable** because real AR aging CSVs often lack it — agent handles the gap per RULES.

### 3.5 Deliberately NOT added

- `Customer` table — all V1.5 demo invoices are one-off; normalize later if needed.
- `DunningAttempt` history table — the existing `Action` + `ActionEvent` relation already gives dunning history via `Invoice.actions`.
- `PaymentTerms` per customer — V2+ concern.

## 4. Agent Layer

### 4.1 New tools in `lib/agent/tools.ts`

Appended to `createFinancialTools(userId)` and returned from the same array so chat and upload both see them.

1. **`scan_ar_aging(dataSourceId?)`** — reads `Invoice` where `status = "open"` AND `(lastDunnedAt is null OR lastDunnedAt < now - 14d)` AND `(snoozedUntil is null OR snoozedUntil < now)`. Computes `daysOverdue = today - dueDate`, buckets into `info (1–14)` / `warning (15–44)` / `critical (45+)`. Returns `{ bucketed: {info, warning, critical}, skipped: [...reasons] }`. Pure read.

2. **`create_ar_actions(actions[])`** — batch insert, same pattern as existing `create_actions`. Each item: `{ invoiceId, severity, headline, detail, driver, dataSourceId }`. Dedupes by `(invoiceId, status = "pending")`. Sets `action.type = "ar_followup"` automatically.

3. **`draft_dunning_email(invoiceId, tone?)`** — `tone ∈ {friendly, firm, escalation}`, defaults to the tone matching the invoice's current bucket. Pulls invoice + customer context, produces an email body, writes it to `action.draftBody` on the matching pending action. Returns the text. Does NOT send anything.

4. **`update_invoice_status(invoiceId, status, snoozedUntil?)`** — writes invoice state transitions. Records an `ActionEvent` on the linked action.

**Tools deliberately NOT added:**
- No `send_email` — out of scope per Option B.
- No `mark_paid` from agent side — paid status comes from a real GL reconciliation, not agent inference. User-only for V1.5.

### 4.2 New skill at `agent/skills/ar-followup/SKILL.md`

Mirrors the shape of `variance-review/SKILL.md`:

- **When to use:** user uploads AR aging CSV, user asks about overdue invoices, proactive AR scan
- **Process:**
  1. Call `scan_ar_aging`
  2. Classify by bucket
  3. Batch-create actions via `create_ar_actions`
  4. For each newly-created action, call `draft_dunning_email` with bucket-appropriate tone
  5. Summarize in ≤150 words leading with the largest overdue balance
- **Output format:** dollar amounts in `$X.XK`, days-overdue as `(N days overdue)`, one headline per bucket at most

### 4.3 RULES.md additions

- "Never send emails. Only draft. The user sends."
- "Respect snooze and cooldown: if `lastDunnedAt` is within 14 days or `snoozedUntil > now`, do not create a new AR action for that invoice."
- "Never mark an invoice as `paid`. That transition is user-only."
- "If `customerEmail` is missing, still create the action but note the gap in `detail` and address the draft to `the customer`."

### 4.4 DUTIES.md additions

- Under "On Data Upload": add an AR branch — "If the CSV is invoice-shaped (has `invoice_number` + `due_date`), run the `ar-followup` skill instead of `variance-review`."
- New section "On AR Scan": same proactive structure as variance, keyed to invoice aging.

### 4.5 SOUL.md addition

One sentence: "When drafting dunning emails, be direct but preserve the customer relationship. Assume the customer is a partner, not an adversary, unless the balance is critical."

## 5. CSV Ingestion + Auto-Detection

### 5.1 Shape detector at `lib/csv/detect-shape.ts` (new)

Single function: `detectCsvShape(headers: string[]): "variance" | "ar" | "unknown"`.

1. **Fast-path regex check** — if headers include any of `invoice`, `due date`, `amount due`, `customer` + numeric ID column, classify as `ar`. If they include `budget` + `actual` (or `plan` + `actual`), classify as `variance`. Tolerant matching like the current regex detector.
2. **LLM fallback** — if fast-path returns `unknown`, call the same Lyzr agent path used by `lib/csv/llm-mapper.ts`. Prompt: *"Given these CSV headers, classify the shape as 'variance', 'ar', or 'unknown' and return JSON."* Reuses the JSON-fence-stripping regex.

### 5.2 New parser at `lib/csv/ar-parser.ts`

- Required fields after mapping: `invoiceNumber, customer, amount, invoiceDate, dueDate`
- Optional: `customerEmail`
- LLM column mapper fallback when headers are non-standard
- Date parsing tolerates `YYYY-MM-DD`, `MM/DD/YYYY`, `DD-MMM-YYYY`
- Unparseable rows collected into `skipped[]` rather than silently dropped
- Returns `{ invoices: ParsedInvoice[], skipped: SkipReason[] }`

### 5.3 Upload route changes (`app/api/upload/route.ts`)

1. After CSV text is loaded, call `detectCsvShape(headers)`
2. Branch:
   - `variance` → existing `parseVarianceCsv` + existing agent prompt (unchanged)
   - `ar` → `parseArCsv`, bulk insert `Invoice` rows via Prisma `createMany` with `skipDuplicates` on the unique index, then call `analyzeArUpload()` (new function in `lib/agent/index.ts`)
   - `unknown` → 400 with the headers the user sent, UI shows "couldn't classify"
3. Same `DataSource` row regardless of shape. `type` stays `"csv"`. `metadata` JSON field stores `{shape: "ar" | "variance"}` — payload change only; `metadata` already exists as `String?`.

### 5.4 `analyzeArUpload` agent prompt

Structurally identical to `analyzeUpload`:

> *"A new AR aging CSV '{fileName}' was just uploaded with {count} invoices (data source ID: {id}). Execute this workflow: (1) call scan_ar_aging with dataSourceId=... (2) compile eligible invoices into a single batch and call create_ar_actions (3) for each newly-created action, call draft_dunning_email with the bucket-appropriate tone (4) provide a brief summary of total overdue balance and top three items. Do not create actions for invoices in cooldown or snoozed."*

### 5.5 Demo seed data

- `public/sample-ar-aging.csv` with ~8 invoices spread across buckets (2 friendly, 3 firm, 2 escalation, 1 ineligible due to recent dunning)
- `scripts/seed-ar.ts` populates both `Invoice` rows and a matching `DataSource`. Runs independent of the variance seed.

### 5.6 Deliberately NOT done

- No "smart upload" UI widget asking the user which shape — detection is fully automatic.
- No per-customer contact resolution — if `customerEmail` is missing it stays missing; agent handles the gap per RULES.

## 6. UI + API Surface

Four targeted changes. No new routes, no new pages, no nav changes.

### 6.1 `components/feed/action-card.tsx` — variant-aware rendering

Card branches on `action.type`:

- `variance | anomaly | recommendation` → existing Approve / Flag / Ask AI / Dismiss (unchanged)
- `ar_followup` → new button row: **Copy & Mark Sent** · **Snooze 7 days** · **Escalate** · **Ask AI** · **Dismiss**

**"Copy & Mark Sent" is a single atomic button:** one click writes `draftBody` to the system clipboard AND POSTs `op: "mark_sent"` to the AR sub-route in the same handler. No two-step confirmation. The rationale is that the user's mental model is "I sent this" — splitting the clipboard copy from the status change would force a second click to complete a mental action that already happened.

**"Snooze 7 days"** is a fixed-7-day button in V1.5. The API accepts a variable `days` parameter (§ 6.3) but the UI exposes only the 7-day default. A snooze picker is a V2 concern.

For `ar_followup`: card is click-to-expand. Clicking the card body (not the buttons) toggles an inline panel showing cached `action.draftBody` in a monospace `<pre>` block. First click on a card whose `draftBody` is null triggers a lazy-load call to the new GET endpoint; subsequent clicks toggle visibility. No edit-in-place for V1.5 — copy, edit in mail client, mark sent.

### 6.2 `components/feed/filter-bar.tsx` — AR filter chip

Filter bar gets one more chip: `All | Critical | Warning | Info | AR`. The AR chip filters by `action.type === "ar_followup"` regardless of severity. Single extra case in the existing filter reducer.

### 6.3 New API route `app/api/actions/[id]/ar/route.ts`

Three POST sub-ops dispatched by `{ op }` body field, all wrapped in a Prisma `$transaction`:

- **`op: "mark_sent"`** → `invoice.status = "sent"`, `invoice.lastDunnedAt = now`, `action.status = "approved"`, insert `ActionEvent` (`pending → approved`)
- **`op: "snooze", days: 7`** → `invoice.status = "snoozed"`, `invoice.snoozedUntil = now + 7d`, `action.status = "dismissed"`, insert `ActionEvent` (`pending → dismissed`)
- **`op: "escalate"`** → `invoice.status = "escalated"`, `action.status = "flagged"`, insert `ActionEvent`, return a fresh draft from `draft_dunning_email(tone: "escalation")`

**Note on the two state machines:** `Invoice.status` and `Action.status` are not the same enum and don't move in lockstep. `Invoice.status` tracks the business entity's lifecycle (`open → sent → snoozed → escalated → paid`). `Action.status` tracks the workflow card's lifecycle (`pending → approved | dismissed | flagged`). On escalate, the invoice moves to its terminal `escalated` state while the action card moves to `flagged` — meaning "this needs human attention beyond the feed's normal resolve loop." The two machines evolve independently; transitions between them are wired in this route handler and nowhere else.

Fourth sub-op on **GET** returns (and lazily generates) the `draftBody` — card click-to-expand hits this on first open. When `draftBody` is null, the route handler calls a shared `draftDunningEmailFor(invoiceId, tone)` helper (extracted from the `draft_dunning_email` tool's body) so the tool and the GET route never drift apart. The helper writes the result to `action.draftBody` before returning, making subsequent GETs cached reads.

**Why a dedicated route:** the generic `/api/actions/[id]/route.ts` only toggles `action.status`; AR transitions touch both `action` and `invoice` atomically and have semantics (snooze, escalate) that don't map to the generic status enum. Keeping them separate avoids a polymorphic mess.

### 6.4 `components/briefing/morning-briefing.tsx` — "Scan AR" button

Second small button next to briefing refresh: **Scan AR**. Calls the same SSE agent endpoint, with a prompt variant asking the agent to run `ar-followup` against all `ready` AR data sources. Absent AR data sources, button disabled with tooltip: *"Upload an AR aging CSV to enable."*

### 6.5 Deliberately NOT added

- No new nav item, no `/ar` route, no AR aging table (rejected during brainstorm — feed is the product)
- No in-card email editor (copy-to-clipboard covers the workflow)
- No bulk dunning campaign — V2
- No customer-level grouping — one invoice = one card; matches variance feed behavior

## 7. State Machine + Error Handling

### 7.1 Invoice state machine

```
open ──(mark_sent)──▶ sent ──(14d elapsed)──▶ open  [re-dunning eligible]
 │                     │
 │                     └──(escalate)──▶ escalated  [terminal for V1.5]
 │
 ├──(snooze)──▶ snoozed ──(snoozedUntil passes)──▶ open
 │
 ├──(escalate)──▶ escalated  [terminal]
 │
 └──(user marks paid, V2 UI)──▶ paid  [terminal]
```

**Critical invariant:** `scan_ar_aging` is the single decision point for eligibility. A `sent` invoice auto-becomes eligible again once 14 days elapse — no cron needed, next scan picks it up naturally.

### 7.2 Action ↔ Invoice over re-dunning

One invoice accumulates multiple historical `Action` rows over time (one per dunning cycle). The **pending** action for a given invoice is always at most one — enforced by `create_ar_actions` dedupe. Closed actions (approved/dismissed/flagged) stay in place as audit history.

### 7.3 Error handling

- **Agent tool errors** — return `{text: "Error: ..."}` on failure, matching existing pattern. Surface through existing `callbacks.onError` pipe in `lib/agent/index.ts`.
- **CSV parse errors** — `parseArCsv` returns `skipped[]` with reasons (`missing_required_field`, `unparseable_date`, `negative_amount`). Upload route surfaces skipped count to UI as a warning banner.
- **LLM column mapper fails** — same fallback as variance: parser returns `{ error: "Could not infer columns", headers }`, route returns 400.
- **Detection misfires** — `unknown` returns 400 with detected headers. If `ar` is mis-tagged for a variance CSV, the AR parser fails on missing required fields and surfaces the same 400. User retries. No "confirm shape" UI step for V1.5.
- **Missing `customerEmail`** — not an error. Action still created, draft generated, detail string notes the gap.
- **Race on state transitions** — all sub-ops run inside Prisma `$transaction`. No partial-state corruption.

## 8. Testing

Matches existing `__tests__/` layout (vitest). No new test harness needed.

1. **`__tests__/lib/csv/detect-shape.test.ts`** — regex fast-path for variance / AR / ambiguous headers. LLM fallback stubbed.
2. **`__tests__/lib/csv/ar-parser.test.ts`** — happy path, non-standard headers (LLM mapper stubbed), skip-row reasons, idempotent re-parse.
3. **`__tests__/lib/agent/ar-tools.test.ts`** — each of the four new tools against a test DB: `scan_ar_aging` bucket math + cooldown + snooze filters, `create_ar_actions` dedupe, `draft_dunning_email` tone-by-bucket defaulting, `update_invoice_status` emits `ActionEvent`.
4. **`__tests__/lib/invoice-state.test.ts`** — state transition matrix, pure function, no DB. Every legal transition succeeds, every illegal one throws.
5. **`__tests__/app/api/actions/ar.test.ts`** — POST with each `op`, assert DB side effects + `ActionEvent` row.
6. **`__tests__/lib/csv/ar-parser-dates.test.ts`** — date format matrix (`YYYY-MM-DD`, `MM/DD/YYYY`, `DD-MMM-YYYY`, garbage). Isolated because date parsing is the most fragile part.
7. **Manual smoke test at `scratch/probe-ar.ts`** — mirrors `scratch/probe.ts`: seeds AR dataset, runs agent, asserts at least N actions created with expected severities. Validates agent-first behavior before shipping, per the project memory's standing rule.

### Out of scope for V1.5 testing

- End-to-end browser tests (no e2e harness in the repo)
- LLM output quality evals (tone of dunning drafts) — judged manually during smoke test
- Load testing

## 10. Out of Scope (tracked for future phases)

- **V2 — Cash forecasting.** Predictive cash visibility. AR timing assumptions feed into this but live in their own skill.
- **V2.5 — Month-end close assistant.** AR reconciliation status becomes part of the close checklist.
- **V3 — Board reporting.** AR aging summary feeds board commentary narrative.
- **Daily scheduler.** Proactive runs without a user click. Needs deployment target decision first (Vercel cron, external scheduler, etc.) — deferred.
- **Google Sheets connector.** Benefits every skill, not just AR. Deserves its own sprint.
- **QuickBooks / Xero API.** Same reasoning. Real AR ingestion via QBO replaces the CSV upload path only when the integration sprint lands.
- **Outbound email delivery.** Resend/SMTP, domain setup, bounce handling, reply tracking. Explicitly rejected for V1.5 — the "review and approve" loop is the product value.
- **Customer entity table.** Normalize when V2 needs per-customer payment history.
- **Bulk dunning campaigns.** Select multiple cards → send all. V2 workflow.
